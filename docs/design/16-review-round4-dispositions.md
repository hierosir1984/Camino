# Camino — Round-4 Review Dispositions (Claude's verification)

> 2026-07-16. Companion to [15-adversarial-review-round4.md](15-adversarial-review-round4.md). All 16 findings verified (CONFIRMED by reasoning/platform knowledge unless noted) and accepted. Resolutions in [17-design-v5.md](17-design-v5.md). Notably, every blocker fix *removes or simplifies* a mechanism rather than adding one — a sign the design is converging on coherence.

## Blockers (6)

| # | Finding | Resolution in v5 |
|---|---|---|
| 1 | Worker-derived code regains privileges via main's own CI after merge; T3 consequence escapes the test tenant | §5.3/§5.5: honest supply-chain framing — a covert payload passing all gates reaches main like any accepted contribution (human or AI); pre-merge containment claims scoped to pre-merge stages; onboarding hardens main CI (minimal token permissions, environment-protected secrets with required reviewers, pinned actions advisory, **persistent self-hosted runners unsupported in v1**); v1 itself needs no privileged main workflows (no deploy scope) |
| 2 | Reverting a fold deletes user-accepted intent; abandoned missions strand it | §3.1 restructure: **the intent ledger lives in the control plane and only user actions mutate it; canon text is a rendered projection** updated at folds. Reverts/abandonment can never delete intent — the next fold re-renders it; status recomputes. (Same projection pattern as status — the architecture becomes uniform) |
| 3 | Exact-SHA evidence incompatible with GitHub PR-merge creating new SHAs | §4.2: **merge-by-push protocol** — the control plane constructs the exact merge commit locally in the pristine clone, validates at that SHA, then fast-forward pushes it (invariant 8 already makes Camino the author of all commits; now including merges). PRs remain the review/audit surface; GitHub marks them merged on push. What is validated is bit-for-bit what lands |
| 4 | Workflow dispatch: run-name is correlation, not idempotency | §4.4: row reclassified to the at-most-once class — no automatic retry on ambiguity; duplicate dispatches tolerated because GitHub CI on worker refs is advisory-only (Camino's runner gates merges); ambiguity durably recorded before any manual retry |
| 5 | A personal PAT cannot be a distinct "Camino-only" push actor | §4.2/§8.3: claim corrected — mission-branch integrity rests on **detection and reconciliation** (ExternalEdit + freshness + merge-by-push, under which a raced manual advance cannot slip into a merge), not on prevention; a distinct **GitHub App identity is a stated prerequisite for any autonomy unlock** (cruise mode), moved out of the PRD registry. PAT + training mode is the v0 posture |
| 6 | Tier-4 plan auto-approval contradicts user-only intent acceptance | §8.3/§1: tier 4 defined as auto-approval **only for plans introducing no new intent** (quick tasks, gap-fix missions executing accepted intent, maintenance); plans carrying new intent always require user confirmation — *by design*: the founding brief keeps "defining product intent" as the human role. The autonomy end state is restated as "interrupted only for genuine escalations and new-intent acceptance" |

## Deferrables (10) — accepted; addressed in v5 text or explicitly registered

7 DoS via intermediate objects and unbounded audit archives (fetch object/size limits + partial-clone filters, archive quotas/retention — §5.1; details registry). 8 External deletion can leave "on-main" presence (adds `suspected-absent` with conservative invalidation — §3.1/§4.5). 9 Idempotency table refinements (push row with intended SHA, merge-by-ref-state, label as (object,label,state), irreversible external effects → recorded ambiguity — §4.4). 10 Lease expiry isn't fencing (lease **generations/fencing tokens**; runner rejects stale-generation writes; kill-confirm before re-grant; missing states added — §4.6). 11 Snapshot polling misses transient A→B→A transitions (stated as an accepted v1 limitation; webhooks later — §4.5). 12 Knowledge promotion authority/validity (candidates visible to same-issue repair attempts; promotion = human batch or deterministic rule-class; entries carry commit/base validity, invalidated on revert — §3.7). 13 The 0/50 bound is invalid under a rolling stopping rule (CONFIRMED — sequential analysis objection is correct; v5 drops the numeric-bound framing entirely: unlocks are policy heuristics with rolling windows, **joint**-distribution guards, continuing labels, instant revocation — §8.3). 14 Evidence ordering across W/C/M/N SHAs (worker-head evidence advisory and bound to W; gating evidence only from Camino-authored candidates; per-item identity under a mission-gate record — §7.2). 15 Mission-terminal accounting misattributes per-issue outcomes (per-issue delivered flag inside mission outcomes; blocked-age penalty defined as provisional, reversed on delivery — §6). 16 Registry completeness (missing items added; GitHub App removed from registry per blocker 5; deployment/post-deploy listed as explicit future scope from the founding brief — §13).
